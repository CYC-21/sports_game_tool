## 1. 列結構慣例

| 分頁 | 第 1 列 | 第 2 列 | 第 3 列起 |
|------|---------|---------|-----------|
| `matches`、`venues`、`teams`、`players`、`player_team_relations`、`matches_pending` | 英文欄位鍵（給程式用） | 中文說明（給人看） | **資料列** |
| `places` | 合併表頭（如 `id 唯一ID`） | **資料列**（無獨立說明列） | 資料列 |

**Apps Script** 已依上表設定：`matches` 等從 **第 3 列** 開始讀資料；`places` 從 **第 2 列** 開始讀。

**`id` 留空**：若 `matches` 某列 `id` 未填，但仍有 `sport`／`venue_id`／`start_time`／`home_team_id`／`away_team_id` 等任一欄有值，API 會自動給 **`AUTO_R{列號}`**（列號為試算表列 1-based），避免整列被略過。建議仍補上正式 `id` 以利維護。`matches_pending` 類似時為 **`PEND_R{列號}`**。

## 2. 各分頁欄位（原始鍵名 → 意義）

### `matches`

| 鍵名 | 說明 |
|------|------|
| `id` | 賽事唯一 ID |
| `sport` | 運動，例：`football`、`baseball` |
| `league_id` | 聯盟 ID |
| `home_team_id` | 主隊 ID → 對應 `teams.id` |
| `away_team_id` | 客隊 ID → 對應 `teams.id` |
| `venue_id` | 場地 ID → 對應 `venues.id` |
| `start_time` | 開賽時間（建議 ISO 8601） |
| `original_time` | 原始時間（延期對照用） |
| `status` | 賽事狀態，見下節「列舉對照」 |
| `verification` | 驗證狀態，見下節「列舉對照」 |
| `last_updated` | 最後更新 |
| `source` | 資料來源 |
| `note` | 備註 |

### `venues`

| 鍵名 | 說明 |
|------|------|
| `id` | 場館 ID |
| `name` | 名稱 |
| `city` | 城市（首頁「📍 台北」類篩選可用此欄） |
| `country` | 國家 |
| `lat`、`lng` | 座標（附近視圖需要） |
| `capacity` | 容量 |
| `verification` | 驗證狀態 |
| `note` | 備註 |

### `places`

表頭為 **`鍵 中文說明`** 合併在一格；API 讀取時會取 **第一個英文 token** 作為鍵（與下列一致）。

| 正規化後鍵名 | 說明 |
|--------------|------|
| `id` | 景點 ID |
| `name` | 名稱 |
| `type` | 類型，例：`food`、`sightseeing` |
| `city` | 城市 |
| `lat`、`lng` | 座標 |
| `description` | 描述 |
| `source` | 來源 |
| `created_at` | 建立時間 |

> **與場館關聯**：目前 `places` 列 **沒有 `venue_id`**。若要實作「某球場附近推薦」，建議新增欄位 `venue_id`（與 `venues.id` 對齊），前端／腳本再以 ID JOIN。在新增前，球場頁只能顯示「同城市」或手動篩選，無法嚴格場館關聯。

### `teams` / `players` / `player_team_relations`

維持表內英文鍵即可；`GET ?resource=teams|players|relations` 會原樣輸出物件陣列（欄位與試算表第 1 列一致）。

### `matches_pending`（投稿）

為 **文字欄**（隊名、場地名、日期時間分欄），與正式表 `matches` 的 **ID 導向** 不同；審核通過後由人工對應成 `teams`／`venues`／`matches` 的 ID 再寫入 `matches`。

| 鍵名（第 1 列） | 說明 |
|----------------|------|
| `id` | 可留空；`doPost` 會自動產生 `P` + 時間戳 + 隨機尾碼 |
| `sport` | 必填（POST 驗證） |
| `league_name` | 選填 |
| `home_team_name` | 必填 |
| `away_team_name` | 必填 |
| `venue_name` | 必填 |
| `date` | 必填（建議與表單 `type="date"` 一致） |
| `time` | 必填（建議與表單 `type="time"` 一致） |
| `source` | 可留空；預設 `web` |
| `submitter` | 選填 |
| `created_at` | 可留空；自動 ISO 時間 |
| `note` | 選填 |

**POST**：對 Web App 的 `exec` URL 使用 `doPost`（見 `apps-script/Code.gs`）。前端見 `web/js/app.js` 投稿表單。

## 3. `GET ?resource=matches` 擴充欄位（JOIN 結果）

除上述 `matches` 原始欄位外，腳本會附加：

| 欄位 | 來源 |
|------|------|
| `home_team_name` | `teams.name`（`home_team_id`） |
| `away_team_name` | `teams.name`（`away_team_id`） |
| `venue_name` | `venues.name`（`venue_id`） |
| `venue_city` | `venues.city` |
| `title` | `home_team_name + " vs " + away_team_name`；若缺隊名則退回 `note` 或 `id` |

前端 **首頁標題列** 建議優先使用 `title`。

## 4. 列舉值對照（Sheet → UI／舊規格）

試算表內目前範例與早期 `資料欄位規格.md` 用語不同，建議前端用對照表，避免寫死英文。

### `matches.status`（賽事）

| Sheet 值 | 建議 UI |
|----------|---------|
| `scheduled` | 已排程／已確認（可視產品用語定一種） |
| `postponed` | 延期 |
| `cancelled` | 取消（若未來有） |
| `tentative` | 未確認（若未來有） |

（若你希望與需求文件的 `confirmed` / `tentative` / `postponed` 完全一致，可在 Sheet 內逐步改值，或只在 Apps Script 裡做 map。）

### `matches.verification` / `venues.verification`

| Sheet 範例 | 建議語意 |
|------------|----------|
| `verified` | 已核實（接近需求里的 `trusted`） |
| `unverified` | 未驗證 |

若之後要與需求完全一致，可統一為：`unverified` | `community` | `trusted`，並在表內改資料。

## 5. 與舊版 `資料欄位規格.md` 的差異（摘要）

| 項目 | 舊規格（草案） | 你目前的 Sheet |
|------|----------------|----------------|
| 賽事主鍵 | `match_id` | `id` |
| 開賽時間 | `starts_at` | `start_time` |
| 標題 | 單欄 `title` | 由 `teams` JOIN 產生 `title` |
| 賽事狀態 | `confirmed` / `tentative` / `postponed` | `scheduled` / `postponed` 等 |
| 場館城市 | `city_label` | `city` |
| places 與場館 | `venue_id` | **尚無**，需加欄若要做場館頁推薦 |

**結論**：以 **本文件 + 試算表第 1 列英文鍵** 為準；前端與新腳本請用 `GET` 回傳的鍵名開發。

### 前端補齊 matches（不改 .gs 時）

若 `…/exec?resource=matches` 筆數少於試算表，`web` 會用 **gviz**（`…/d/{SHEET_ID}/gviz/tq?sheet=matches&tqx=out:json;responseHandler=…`）讀取並與 `teams`、`venues` 於瀏覽器內 JOIN，邏輯見 `web/js/gviz.js`、`web/js/enrich.js`。試算表連結權限需允許匿名檢視。
